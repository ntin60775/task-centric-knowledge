# Черновик CLI UI/UX для `task-centric-knowledge`

## Паспорт

| Поле | Значение |
|------|----------|
| ID задачи | `TASK-2026-0008` |
| Артефакт | `cli-ux-draft` |
| Версия | `3` |
| Дата обновления | `2026-04-12` |
| Статус | `черновик Этапа 3 для архитектурного контракта` |

## 1. Цель

Зафиксировать минимальный операторский CLI-контракт для `task-centric-knowledge`, чтобы skill давал не только внутреннюю модель и helper-скрипты,
но и предсказуемый пользовательский опыт для запросов:

- `status`
- `current-task`
- `task show`
- `doctor deps`
- `migrate cleanup`

Этот артефакт описывает именно UX-контракт.
Он не требует в рамках `TASK-2026-0008` немедленной реализации конкретного бинаря, wrapper-а или subcommand tree.

## 2. Принципы интерфейса

- По умолчанию вывод компактный, читаемый в терминале и пригодный для ежедневной работы.
- Любой вывод по задаче обязан показывать не только `TASK-ID`, но и `краткое имя` плюс человекочитаемое описание.
- Локальный сценарий без remote, без `gh` и без `glab` считается нормальным, а не деградированным режимом.
- Ошибки и предупреждения должны различать `blocking` и `non-blocking`.
- Любой cleanup после миграции всегда идёт через `plan -> confirm`, без молчаливого удаления.

## 3. Командная поверхность

### Канонические intent-имена

| Intent | Канонический вызов | Назначение |
|--------|---------------------|------------|
| Общий статус | `status` | Сводка по knowledge-системе и задачам репозитория |
| Текущая задача | `current-task` | Сжатое состояние активной задачи и её подзадач |
| Карточка задачи | `task show <selector>` | Детальный просмотр конкретной задачи |
| Диагностика зависимостей | `doctor deps` | Проверка обязательных, условно-обязательных, опциональных и `not-applicable` зависимостей окружения |
| План cleanup после миграции | `migrate cleanup plan` | Dry-run устаревших данных после миграции |
| Подтверждённый cleanup | `migrate cleanup confirm` | Явное удаление только после показа плана |

### Замечание по транспорту

- Финальный transport layer может быть оформлен как отдельный CLI entrypoint, как subcommands существующего helper-а или как thin wrapper.
- Независимо от транспорта, смысл команд и обязательные поля вывода должны оставаться одинаковыми.

## 4. Форматы вывода

### Общий формат

- Формат по умолчанию: компактный `pseudo-TUI`.
- Полный подробный режим допускается только по явному запросу пользователя.
- Машинно-значимые поля, пути, URL, команды и ID не смешиваются с длинной прозой.

### Обязательный заголовок задачи

Любой вывод о задаче обязан начинаться с трёх элементов:

1. Полный `TASK-ID`
2. `Краткое имя`
3. Человекочитаемое описание

Минимальный образец:

```text
TASK-2026-0008 · task-centric-knowledge-roadmap-vnext
Стратегический пакет vNext для развития task-centric-knowledge
```

### Источник полей

- `TASK-ID`, `Краткое имя` и `Человекочитаемое описание` берутся из `task.md`.
- `registry.md` используется только как навигационный кэш и не является источником истины для сводки задачи.
- Если описание в `registry.md` отсутствует или расходится с `task.md`, система должна показать предупреждение о drift-е, но не должна подменять данные из `task.md`.
- Для legacy-задач без поля `Человекочитаемое описание` допустим временный fallback на первый абзац раздела `## Цель` с обязательным migration-warning.

### Общие поля task-oriented read-модели

Любая команда, которая показывает конкретную задачу или список кандидатов задач, обязана опираться на один и тот же минимальный набор полей:

- `TASK-ID`;
- `Краткое имя`;
- `Человекочитаемое описание`;
- `Статус`;
- `Текущий этап` или явно названный следующий шаг;
- `Ветка`, если она однозначно определена;
- `Блокеры / ручной остаток`, если они влияют на следующее действие.

Если команда не может надёжно вывести одно из этих полей, она должна показать предупреждение о неполноте данных, а не молча скрывать пробел.

## 5. Отчёт `status`

### Назначение

Показать состояние knowledge-системы репозитория в одном экране.

### Что обязано быть в выводе

- активная ветка;
- наличие и состояние `knowledge/`;
- текущая активная задача, если она однозначно определяется;
- количество задач по ключевым статусам;
- список задач с приоритетом `высокий`, которые не завершены;
- задачи `на проверке`;
- наличие открытых delivery units;
- диагностические предупреждения:
  `нет registry`, `битый managed-block`, `несинхронная ветка`, `неконсистентные статусы`.

### Что важно для UX

- `status` должен отвечать быстро и не требовать чтения всех task-файлов подряд вручную.
- Если активная задача не определяется однозначно, это должно быть показано как явный warning, а не потеряно в деталях.

### Минимальная структура экрана

- блок `Repository / branch`;
- блок `Knowledge health`;
- блок `Current task` или `Ambiguous current task`;
- блок `Task counters`;
- блок `Open delivery units`, если они есть;
- блок `Warnings / blockers`.

## 6. Сводка `current-task`

### Назначение

Дать короткий и пригодный для продолжения работы ответ на вопрос:
`над чем я сейчас работаю и что дальше`.

### Что обязано быть в выводе

- заголовок задачи:
  `TASK-ID + краткое имя + человекочитаемое описание`;
- статус и текущий этап родительской задачи;
- активная ветка;
- список ключевых подзадач с их статусами;
- если есть delivery units, то их краткая таблица:
  `Unit ID`, `Host`, `Статус`, `URL`;
- следующий ожидаемый шаг;
- важные blockers или ручные проверки, если они ещё остались.

### Поведение на неоднозначности

- если текущая задача не одна, выводить краткий список кандидатов и явно помечать состояние как `неоднозначно`;
- если активной задачи нет, так и писать:
  `Активная задача не определена`.
- если есть несколько кандидатов с одинаковой степенью уверенности, команда не должна выбирать одну задачу молча.

## 7. Карточка `task show`

### Назначение

Показать детальную, но всё ещё читаемую карточку выбранной задачи.

### Что обязано быть в выводе

- `TASK-ID + краткое имя + человекочитаемое описание`;
- `Parent ID`, статус, приоритет, ветка;
- цель и текущий этап;
- связанные файлы:
  `task.md`, `plan.md`, `sdd.md`, важные артефакты;
- подзадачи и их статусы;
- publish-контур, если он есть;
- короткий verify-блок:
  что покрывается автоматикой, что осталось вручную.

### Поддерживаемые селекторы

- точный `TASK-ID`;
- `current`;
- допустимо в будущем: уникальное краткое имя или slug.

### Ошибки выбора задачи

- если `current` неоднозначен, команда возвращает warning и список кандидатов вместо произвольного выбора;
- если селектор не найден, команда обязана явно показать, по каким правилам выполнялся поиск;
- если найдено несколько задач по нестрогому селектору, команда должна потребовать уточнение, а не показывать случайное совпадение.

## 8. `doctor deps`

### Назначение

Проверить, готово ли окружение для install/check/upgrade/publish сценариев.

### Минимальные классы зависимостей

- обязательные:
  `python3`, `git`, доступность файлов skill-а, доступность `knowledge/` и `AGENTS.md`-контуров;
- условно-обязательные:
  инструменты, реально нужные для выбранного режима или профиля;
- опциональные:
  `gh`, `glab`, auth к forge-хостингам, если пользователь не использует соответствующий publish-layer.
- `not-applicable`:
  зависимости и интеграции, которые не относятся к активному режиму работы и не должны трактоваться как ошибка.

### Что обязано быть в выводе

- для каждой зависимости статус:
  `ok`, `missing`, `misconfigured`, `optional`, `not-applicable`;
- короткое объяснение, зачем она нужна;
- точная подсказка по исправлению;
- различение между тем, что блокирует `core/local mode`, и тем, что блокирует только `publish/integration`.

### Важный UX-инвариант

- отсутствие `gh` или `glab` не должно выглядеть как поломка всего skill-а, если пользователь работает в `core/local mode`.

### Минимальный формат строки зависимости

Для каждой зависимости вывод должен включать:

- имя зависимости;
- класс зависимости;
- статус `ok`, `missing`, `misconfigured`, `optional` или `not-applicable`;
- слой, который она блокирует: `core/local mode` или `publish/integration`;
- короткую подсказку по исправлению или причину, почему зависимость не требуется.

## 9. `migrate cleanup`

### Назначение

Безопасно завершать миграцию со старых версий skill-а без молчаливого удаления старых данных.

### Двухшаговый flow

1. `migrate cleanup plan`
2. `migrate cleanup confirm`

### Что обязано делать `plan`

- находить устаревшие managed-файлы, backup-артефакты и старые служебные контуры;
- показывать абсолютные пути;
- показывать `TARGETS` и краткую сводку по ним;
- показывать `TARGETS`, `TARGET_COUNT`, `COUNT`;
- показывать точную команду cleanup или эквивалентный список действий;
- различать:
  `безопасно удалить`,
  `нужно сохранить`,
  `нужна ручная проверка`.

### Что обязано делать `confirm`

- повторно показывать план cleanup;
- требовать явного подтверждения;
- не расширять scope удаления по сравнению с показанным планом;
- завершать cleanup только после подтверждения пользователя.

### Scope-lock инварианты

- `plan` и `confirm` обязаны оперировать одним и тем же набором `TARGETS`;
- если между шагами изменились `TARGET_COUNT`, `COUNT` или точная команда cleanup, система обязана остановиться и заново показать план;
- если cleanup нельзя посчитать надёжно, `confirm` недопустим;
- для любого рекурсивного cleanup scope должен раскрываться до уровня, достаточного для надёжного подсчёта удаляемых объектов.

## 10. Обязательные ошибки и предупреждения

Система должна иметь человекочитаемые сообщения минимум для случаев:

- `knowledge/` отсутствует;
- `registry.md` отсутствует или повреждён;
- `task.md` и `registry.md` расходятся;
- текущая задача определяется неоднозначно;
- publish-host инструмент отсутствует, но requested action требует его;
- migration cleanup нельзя посчитать надёжно;
- в `task.md` нет человекочитаемого описания.

### Контракт уровней серьёзности

- `blocking` — команда не может дать корректный результат и должна остановиться;
- `non-blocking` — команда может продолжить работу, но обязана показать warning и объяснение;
- `migration-warning` — временный fallback допустим только как переходный режим и должен сопровождаться явной подсказкой, что требуется обновление task-данных.

## 11. Что это означает для следующих delivery-задач

- Нужен отдельный query/reporting layer, а не только mutate/helper-команды.
- `status`, `current-task` и `task show` должны опираться на единый контракт read-модели.
- `doctor deps` должен стать частью install/upgrade governance.
- `migrate cleanup` обязан наследовать строгий confirm-flow и dry-run discipline.
