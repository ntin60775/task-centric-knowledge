## Цель

Зафиксировать и проверить безопасный сценарий обновления старой версии `task-centric-knowledge` на новую без потери уже созданных task-данных, без дубликатов managed-структур и с однозначной фиксацией перехода в локальном git.

## Границы

### Входит

- Правила безопасного обновления и перехода между версиями skill-а.
- Явное разделение перехода на локальные git-шаги и отдельные переходные commit-ы.
- Правило про предложение `push` только при наличии связанного удалённого репозитория.
- Тесты на сохранность уже созданных task-данных и отсутствие дубликатов managed-блоков.

### Не входит

- Автоматический `push` или открытие PR.
- Миграция произвольных внешних project-specific данных, не относящихся к managed-ресурсам skill-а.

## Контекст

- источник постановки: ожидание пользователя, что обновление старого навыка на новую версию должно проходить без регрессий и с ясно выделенным моментом перехода в git;
- связанная область: upgrade-процесс `task-centric-knowledge`, managed-файлы и локальный git-поток;
- ограничения и зависимости: приоритет у локального git; push можно предлагать только когда реально обнаружен связанный удалённый репозиторий;
- основной контекст сессии: `подзадача внутри TASK-2026-0004`.

## Стратегия проверки

### Покрывается кодом или тестами

- Unit-тесты installer-а на idempotent update и сохранность пользовательских task-данных.
- Проверка helper-а на обнаружение связанного удалённого репозитория.
- `markdown-localization-guard`.
- `git diff --check`.

### Остаётся на ручную проверку

- Оценка того, что предложенный transition-playbook действительно удобен для реального обновления skill-а в других репозиториях.

## Критерии готовности

- В skill-е есть явный playbook безопасного обновления старой версии на новую.
- Есть правила, как разносить переход по локальному git и как выделять момент перехода отдельным commit-ом.
- Навык не предлагает `push`, если у локального репозитория нет связанного remote.
- Тесты подтверждают отсутствие потерь project data и дубликатов managed-структур.
