## 0. Инварианты и verification matrix

### Полный invariant set

- `INV-01`: стратегия должна содержать явное решение по вопросу `эволюция / redesign / fork`; выбранный вариант должен быть назван как `redesign ядра без full rewrite`, а не оставлять равновероятные варианты без выбора.
- `INV-02`: стратегия должна опираться на текущую реальность skill-а в этом репозитории, включая уже реализованные capability и lessons learned из `TASK-2026-0007`.
- `INV-03`: стратегия должна включать конечную целевую форму `vNext` и промежуточные фазы с наблюдаемыми результатами.
- `INV-04`: внешние аналоги должны рассматриваться как источники идей или потенциального заимствования только после сравнения по официальным источникам.
- `INV-05`: итог не должен толкать систему в полный rewrite без доказанных причин и не должен рекомендовать full-fork при отсутствии близкого полного аналога.
- `INV-06`: дорожная карта должна уменьшать change amplification и архитектурное разрастание, а не консервировать их.
- `INV-07`: стратегия должна включать явный CLI query/reporting contract, а не только внутренние helper-механики.
- `INV-08`: любой вывод по задаче в будущем CLI должен требовать связку `TASK-ID + краткое имя + человекочитаемое описание`.
- `INV-09`: install/upgrade governance должен включать `doctor deps` с разделением обязательных, условно-обязательных, опциональных и `not-applicable` зависимостей.
- `INV-10`: cleanup после миграции не должен происходить молча и обязан идти через `plan -> confirm`.
- `INV-11`: `task.md` должен оставаться единственным источником истины для сводки задачи, а `registry.md` должен быть только навигационным cache.
- `INV-12`: `vNext` должен иметь DDD-карту контекстов с ограниченными контекстами и границами владения.
- `INV-13`: все доказательные артефакты задачи должны жить внутри каталога задачи или иметь явную ссылку из него.
- `INV-14`: дорожная карта уровня skill-а не должна конкурировать с дорожной картой задачи и обязана быть синхронизированным снимком с тем же набором внешних источников и тем же стратегическим выводом.
- `INV-15`: имя `task-centric-knowledge` сохраняется, пока центром продукта остаётся локальное знание задачи и её жизненный цикл.

## 1. Проблема и цель

### Проблема

`task-centric-knowledge` развивался серией правильных локальных итераций:

- `TASK-2026-0004` добавила git-aware жизненный цикл, маршрутизацию задач и безопасность обновления;
- `TASK-2026-0006` добавила testing contract и `verification-matrix`;
- `TASK-2026-0007` добавила publish-flow и delivery units и уже прошла GitHub smoke-gate.

Из локального аудита видно, что система уже превратилась из набора шаблонов в workflow-framework. Самый явный технический симптом:

- `skills-global/task-centric-knowledge/scripts/task_workflow.py` содержит около `1893` строк и смешивает доменную модель, markdown I/O, git orchestration, publish-flow и host adapters;
- нормативные правила размазаны по `SKILL.md`, `references/*.md`, managed-блокам, `knowledge/tasks/README.md`, task-шаблонам и тестам;
- после ввода поля `Человекочитаемое описание` helper продолжал обновлять `registry.md` через `--summary` и fallback от `## Цель`, из-за чего канонический источник истины не доходил до навигационного cache;
- `skills-global/task-centric-knowledge/agents/openai.yaml` уже не полностью отражает реальный scope skill-а.

Если продолжать добавлять фичи без явной стратегической рамки, skill рискует стать слишком общим и трудно обновляемым.

### Цель

Зафиксировать для `task-centric-knowledge` стратегию `vNext`, при которой:

1. полный rewrite не запускается без доказанной необходимости;
2. full-fork от внешнего решения не выбирается при отсутствии близкого аналога;
3. выбран `redesign ядра без full rewrite`;
4. дальнейшее развитие идёт фазами: стабилизация -> контракт ядра -> модульная декомпозиция -> расширения;
5. каждый следующий слой развития открывается только после прохождения проверочных ворот.

## 2. Архитектура и границы

- Источник истины по текущему состоянию skill-а: `skills-global/task-centric-knowledge/` и live-контур `knowledge/tasks/`.
- Источник истины по стратегии этой задачи: `artifacts/strategy-roadmap.md`.
- Изолированное evidence Этапа 1: `artifacts/stage-1-audit.md`.
- Стратегия не меняет автоматически сам skill, а определяет, какие отдельные delivery-задачи нужно открывать дальше.
- Конечная целевая форма системы: модульный `task operating system for repositories`, а не просто installer и не абстрактная universal memory platform.

- Допустимо: заимствовать идеи из внешних систем без отказа от собственного канонического `task.md / plan.md / sdd.md`.
- Недопустимо: делать `task-centric-knowledge` vendor-locked memory-слоем, завязанным на одну IDE или hosted platform.
- Недопустимо: смешивать стратегическую задачу с фактическим большим refactor-изменением skill-а в одном шаге.

## 4.1. DDD-карта контекстов

### `Task Core`

Основной ограниченный контекст.

- Корень агрегата: `Task`.
- Внутренние сущности: `Subtask`, `Delivery Unit`, `Verification Matrix`, `Task Artifact`, `Decision`, `Worklog Entry`, `Handoff`.
- Объекты-значения: `Task ID`, `Task Slug`, `Task Status`, `Branch Ref`, `Publication State`, `Evidence Ref`.
- Источник истины: `task.md` и локальный каталог задачи.
- `registry.md` не является корнем агрегата и не владеет сводкой задачи.

### `Read Model / Reporting`

Прикладной контур, производный от `Task Core`.

- Команды: `status`, `current-task`, `task show`.
- Пишет только диагностические предупреждения и отчёты, если это явно задано будущей реализацией.
- Не меняет доменную модель и не становится источником истины.

### `Publish Integration`

Интеграционный ограниченный контекст.

- Отвечает за адаптеры forge-хостингов и publish-состояние delivery unit.
- Не меняет `Task Core` напрямую в обход канонических переходов состояний.
- GitHub/GitLab-специфика не должна протекать в core.

### `Memory`

Расширяемый контур проектной памяти.

- Может добавлять memory lanes поверх локального знания задачи.
- Не конкурирует с `task.md` за роль источника истины.
- Любая promoted memory должна ссылаться на локальный источник задачи, если она выводится из задачи.

### `Packaging / Governance`

Прикладной и инфраструктурный ограниченный контекст.

- Отвечает за install, check, upgrade, doctor и cleanup после миграции.
- Не удаляет project data без явного cleanup-плана и подтверждения.

### `Profiles`

Расширяемый ограниченный контекст.

- Добавляет профильные формулировки и шаблоны.
- Не меняет семантику ядра задачи, статусов и владения.

## 4.2. Минимальный контракт `vNext-core`

Этап 3 фиксирует не реализацию, а минимальный нормативный контракт ядра.
Этот контракт обязан быть достаточно коротким, чтобы его можно было использовать как первичный источник при будущей модульной декомпозиции.

### Что считается частью ядра

- канонические файлы задачи:
  `task.md`, `plan.md`, `sdd.md` и каталог `artifacts/`;
- доменные сущности:
  `Task`, `Subtask`, `Delivery Unit`, `Verification Matrix`, `Task Artifact`, `Decision`, `Worklog Entry`, `Handoff`;
- объекты-значения:
  `Task ID`, `Task Slug`, `Task Status`, `Branch Ref`, `Publication State`, `Evidence Ref`;
- каноническая task summary:
  `TASK-ID`, `Краткое имя`, `Человекочитаемое описание` из `task.md`;
- базовые правила маршрутизации, git-инварианты и владение локальными доказательными артефактами.

### Что обязано оставаться производным слоем

- `registry.md`, `status`, `current-task` и `task show` как навигационный кэш и read-модель поверх `Task Core`;
- forge-адаптеры и publish-flow как `Publish Integration`;
- memory lanes и внешние knowledge-надстройки как `Memory`;
- install/check/upgrade/doctor/cleanup как `Packaging / Governance`;
- профильные формулировки как `Profiles`.

### Обязательные правила владения

- `task.md` остаётся единственным источником истины для task summary и семантики статуса;
- `registry.md` не может владеть описанием задачи и не может переопределять поля из `task.md`;
- `Publish Integration`, `Memory` и `Profiles` не могут менять `Task Core` в обход канонических переходов состояний;
- `Packaging / Governance` не удаляет project data без явного `plan -> confirm`;
- все доказательные артефакты задачи принадлежат локальному каталогу задачи, если только внешний объект явно не привязан ссылкой и объяснением.

### Что Этап 3 сознательно не выбирает

- транспорт CLI: отдельный entrypoint, subcommand tree или thin wrapper;
- фактический file split runtime-модулей;
- реализацию memory lanes, forge parity и package/lock governance;
- точную delivery-декомпозицию Этапа 4.

## 7. Критерии приёмки

1. В задаче есть явное решение, нужно ли продолжать текущий путь, делать redesign или искать fork.
2. В стратегическом артефакте есть сравнение с внешними системами по официальным источникам.
3. Конечная цель `vNext` описана как проверяемая архитектурная форма, а не как абстрактное пожелание.
4. Есть фазовая дорожная карта с промежуточными результатами и воротами перехода.
5. Стратегия учитывает текущие локальные хвосты и не отрывается от реального состояния skill-а.
6. В стратегии есть минимальный операторский CLI-контракт для запросов, отчётности и диагностики миграции.
7. Есть DDD-карта контекстов с ограниченными контекстами и границами владения.
8. Helper синхронизирует `registry.md` от `Человекочитаемого описания` из `task.md`, а legacy-сценарий без этого поля не ломается.
9. Строка `TASK-2026-0008` в `knowledge/tasks/registry.md` совпадает с канонической сводкой из `task.md`.
10. Хранение доказательных артефактов поднято в инвариант ядра локального каталога задачи.

## 8. Стоп-критерии

1. Если для рекомендуемой внешней системы нельзя показать покрытие ключевых требований `task-centric-knowledge`, вариант full-fork должен быть снят.
2. Если roadmap не уменьшает change amplification и не даёт границу ядра, задачу нельзя считать успешно выполненной.
3. Если дальнейшее развитие снова требует добавлять фичи в монолит без промежуточной стабилизации, roadmap нужно пересмотреть до начала реализации.
4. Если read-модель или слой памяти начинают становиться источником истины вместо `task.md`, развитие нужно остановить и вернуть владение в `Task Core`.
