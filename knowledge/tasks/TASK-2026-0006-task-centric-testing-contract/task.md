## Цель

Доработать `task-centric-knowledge` так, чтобы тестовый контракт задачи по умолчанию требовал только реально исполнимых и допустимых правилами проекта автоматических проверок по всем затронутым контурам, а все оставшиеся ручные проверки собирались в единый итоговый список на уровне общей задачи.

## Границы

### Входит

- Обновить документацию skill `task-centric-knowledge`.
- Обновить task-шаблоны и связанные правила knowledge-системы.
- Явно зафиксировать приоритет автоматических проверок для бэкенда, фронтенда и остальных затронутых компонентов.
- Явно зафиксировать единый итоговый список ручных проверок на уровне общей задачи.
- Обновить тесты skill-а под новый контракт.
- Синхронизировать локальный `knowledge/`-контур репозитория с обновлёнными шаблонами.

### Не входит

- Реализация конкретного продуктового тестового раннера вне навыка `task-centric-knowledge`.
- Автоматическое исполнение произвольных e2e/regression-наборов во внешних репозиториях.

## Контекст

- источник постановки: ожидание пользователя, что ручные тесты не должны быть размазаны по артефактам задачи, а автоматические проверки должны включать только реально исполнимый и допустимый правилами проекта набор;
- связанная область: skill `task-centric-knowledge`, knowledge-шаблоны задач и правила ведения проверки;
- ограничения и зависимости: новый контракт не должен ломать существующий task-flow и должен оставаться универсальным для разных типов проектов;
- основной контекст сессии: `текущая задача`
